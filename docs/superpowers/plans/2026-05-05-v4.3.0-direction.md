## A. 候选 workstreams（按 leverage 排序）

### A1. 前端 mega-component 收敛（结构延续）

非 v4.2.0 hot-spot 但已超 1000 行的 5 个文件：

| 路径 | 行数 | 切分方向 |
|---|---:|---|
| `frontend/src/components/cross-market/CrossMarketResultsSection.js` | 1331 | 摘要 / 表格 / 诊断 / 研究剧本入口分子组件 |
| `frontend/src/components/research-playbook/playbookViewModels.js` | 1314 | view model 按 playbook 类型分文件 |

| `frontend/src/components/research-workbench/workbenchUtils.js` | 1005 | utility 按主题拆 |

**优势**：模式与 v4.2.0 后端 god-module 拆分一致（≤600 cap、零行为变更位移、Jest 锁测试），可复用经验。
**劣势**：前端组件渲染状态多、UI 测试覆盖弱（多数走 e2e 兜底），切分风险高于后端。
**建议**：优先 `playbookViewModels.js` 与 `crossMarketRecommendations.js`（util 层、纯函数为主，最容易切）；UI 组件次之。

### A2. 测试覆盖率提升（55% → 65%）

CI 当前 `--cov-fail-under=55`。覆盖率门槛偏低，意味着大量代码路径无回归保障。
v4.2.0 删了不少测试（与 endpoint 一起删），需要把节省的 budget 投到真正核心
的路径覆盖上。

**方法**：
- 用 `coverage report --skip-covered` 找空白
- 优先补 `src/analytics/{pricing_gap_analyzer,industry_analyzer}.py`、
  `src/backtest/backtester.py`、`backend/app/services/quant_lab.py` 这种
  核心引擎代码
- 不补 `src/data/providers/*` 的网络路径（已在 manual / integration suite）

**优势**：直接降低重构 / 重构后 regression 成本；CI 信号更强。
**劣势**：补测试对功能交付的体感价值低；纯工程债。

### A3. 生产安全：PG → SQLite 静默降级修正

`backend/app/core/persistence/_manager.py` 的 connect_postgres 失败会静默
fallback 到 SQLite，可能导致：
- 数据库写入到了"错误的"目标（SQLite 而非声明的 PG）
- 启动后没人发现，等到第一次重启才暴露

**当前行为**（session 1 评估时已识别）：
```python
# 简化伪代码
try:
    pg = connect_postgres(url)
    return pg
except ConnectionError:
    logger.warning("PG unreachable, falling back to SQLite")
    return connect_sqlite(SQLITE_PATH)
```

**目标**：在 `ENVIRONMENT != development` 时，PG 连接失败应 fail-fast（与
auth secret 守卫一致），让 systemd / docker-compose 重启循环显形。

**优势**：小改动（约 20 行），实质性的生产安全提升。
**劣势**：需要本地有真实 PG 才能测试 happy path；可能引发其他依赖该静默 fallback 的代码暴露。

### A4. 维护脚本群审计

14 个 `refresh_*` / `verify_*` / `inspect_*` / `check_*` 脚本有完整 docstring

**两个选项**：

**优势**：结束第三方对脚本目录的混淆。
**劣势**：低价值，可能引发手动维护工作流崩溃。

### A5. 性能 / 缓存层评估

**可能的具体工作**：

**优势**：用户可见的速度改进。
**劣势**：需要先有可量化的 baseline；性能测试基础设施不齐。

## B. 主动监控的非 workstream 风险

下面这些是 session 1 / session 2 评估期间识别但未在 v4.2.0 范围内处理的，
需要保留视野：

1. **SQLite RLock 串行化**：`_records.py` 跨连接生命周期持锁，并发上限约
   2-5 用户。生产部署应用 PG。
2. **Dual-track schema 迁移**：Alembic（PG）与 `_manager.py` 内联 bootstrap
   （SQLite）需双轨更新；遗漏一边会让 SQLite 用户与 PG 用户 schema 漂移。
3. **CI 取消链 pattern**：rapid-push 时 GitHub 默认取消 in-flight job，导致
   多个连续 commit 实际上没有完整 CI 验证。建议显式等 CI 跑完再 push 下一个，
   或者用一个累积分支再 squash-merge。
4. **覆盖率 55% 阈值偏低**：实际线下 `coverage report` 可能高得多，但 CI 门
   不强迫。应做一次摸底再考虑抬高。
